# Migrating to plugboy v1 (tsdown)

plugboy's internal bundler moved from **tsup (esbuild)** to **tsdown (rolldown)**.
This changes the workspace config schema, the plugin-authoring API, and a few
output details.

- **Affected packages:** `@fastkit/plugboy` (`0.3.x` → `1.0`) and every
  `@fastkit/plugboy-*-plugin` (bumped to their next major together).
- **Who needs to act:** anyone who builds packages with plugboy, and anyone who
  authors a custom plugboy plugin.

## Highlights (new in v1)

- **Vue SFC support** via the new `@fastkit/plugboy-vue-plugin`
  (`unplugin-vue/rolldown`). When `vue` is a dependency it also switches the
  declaration compiler to `vue-tsc` automatically. This was impractical under
  the old esbuild pipeline.
- `@fastkit/plugboy-vue-jsx-plugin` is reimplemented on `unplugin-vue-jsx/rolldown`.
- New declaration options: `dts.compiler` (`'tsc'` | `'vue-tsc'` | custom fn) and
  `dts.ignoreCompilerErrors`.
- Build-time env constants `__PLUGBOY_DEV__` / `__PLUGBOY_STUB__` you can use in
  your source to guard dev-only code/logs. In published output `__PLUGBOY_DEV__`
  evaluates the **consumer's** runtime (`NODE_ENV` / `import.meta.env.DEV`). See
  [Env constants](../env-constants.md).

## Breaking changes

### 1. `external` / `noExternal` / `replaceNodeEnv` → `deps`

These tsup-era options are gone. Use tsdown's `deps`:

```ts
// before
defineWorkspaceConfig({
  external: ['vue'],
  noExternal: ['some-pkg'],
});

// after
defineWorkspaceConfig({
  deps: {
    neverBundle: ['vue'], // was `external`
    alwaysBundle: ['some-pkg'], // was `noExternal`
  },
});
```

`replaceNodeEnv` has no direct replacement and was removed. From a plugin, use
the new `ctx.mergeExternals()` / `ctx.mergeNoExternals()` helpers in
`setupWorkspace` to append to `deps` dynamically.

### 2. Output: declaration files are now `.d.mts`

To match the `.mjs` ESM output, type definitions are emitted as `.d.mts`.
plugboy regenerates each package's `package.json` `exports`/`types`, so a rebuild
updates them automatically. Fix any hand-written references to `*.d.ts`.

### 3. Dependency subpath rename

- `@fastkit/plugboy/dependencies/tsup` → `@fastkit/plugboy/dependencies/tsdown`
- New: `@fastkit/plugboy/runtime-utils`

### 4. `@fastkit/plugboy-vanilla-extract-plugin/css` → `@fastkit/vanilla-extract-utils`

The authoring helpers (`defineLayerStyle`, `createGlobalTheme`, …) moved out of
the build plugin into a standalone, bundler-agnostic package.

```ts
// before
import { defineLayerStyle } from '@fastkit/plugboy-vanilla-extract-plugin/css';
// after
import { defineLayerStyle } from '@fastkit/vanilla-extract-utils';
```

Add `@fastkit/vanilla-extract-utils` to your dependencies. It declares
`@vanilla-extract/css` as a **peer dependency**, so also ensure `@vanilla-extract/css`
is installed (it must be a single instance shared with your own vanilla-extract
usage). The build plugin (`@fastkit/plugboy-vanilla-extract-plugin`) no longer
exposes `/css`.

## Custom plugin authors

### `esbuildPlugins` → push tsdown/rolldown plugins

`Plugin` now extends tsdown's `TsdownPlugin`. Instead of `esbuildPlugins`, push a
rolldown/tsdown plugin onto `ctx.plugins` in `setupWorkspace`.

```ts
// before
definePlugin({
  name: 'my-plugin',
  esbuildPlugins: [(workspace) => SomeEsbuildPlugin(options)],
});

// after
definePlugin({
  name: 'my-plugin',
  hooks: {
    setupWorkspace(ctx) {
      ctx.plugins.push(SomeRolldownPlugin(options));
    },
  },
});
```

### `onSuccess` plugboy hook removed

The `onSuccess(builder, outputFiles)` plugboy hook is gone. For post-build work,
use tsdown's `onSuccess` config option, or a pushed plugin's
`writeBundle` / `generateBundle`.

### `setupWorkspace` gains a second argument

`setupWorkspace(ctx, getWorkspace)` — `getWorkspace: () => PlugboyWorkspace | undefined`
is available when needed.
